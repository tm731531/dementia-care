# 健康飲品成分分析 — 營養成分並排比較工具

> 從給女兒挑奶粉開始，後來發現失智媽媽的醫療營養品也需要，再後來藥師朋友說他也有一模一樣的需求。
> 所以做一個純前端比較網站——在貨架前掏手機 3 秒看出差異。

資料是**拍包裝標籤 → AI 辨識擷取 → 手動填 JSON**，不是爬蟲。

🔗 **網站**：<https://tm731531.github.io/dementia-care/health-drinks/docs/>

## 起源與演化

這個工具的使用者從一個變三個，需求都一模一樣：

1. **🍼 Tom 自家（起源）**：為 2023 年出生的女兒挑奶粉。桂格、惠氏等各品牌嬰幼兒配方的蛋白質、鐵、DHA、乳清蛋白比例排法都不一樣，肉眼比不出來
2. **🧓 失智媽媽**：擴展到醫療營養品選購（亞培安素 HMB、葡勝納 SR、費森尤斯倍速益、補體素、三多補體健、倍力素癌症病人配方⋯）
3. **💊 藥師朋友**：他在藥局遇到**一模一樣的需求**——客戶問「哪罐好」要當場比對，肉眼掃成分表超容易漏行。現在他也用這個工具

這解釋了為什麼 `drinks.js` 同時涵蓋**嬰幼兒配方**、**醫療營養品**、**高蛋白補充品**、**雞精**等跨類別——每一類都對應到上面某一個使用者的需求。

---

## 真正的兩個痛點（不是「方便查詢」這種 generic 理由）

#### 1. 購買端：排序、單位不統一，客戶站在面前比不出來

各品牌的營養成分表**排列順序不一樣**、**單位也不一樣**：
- 有的先蛋白質、有的先熱量
- 有的 per 份、有的 per 100ml
- 熱量有時 kcal 有時 kJ，礦物質 mg/mcg 混用
- 微量元素（碘、硒、鉻、維生素 B12）有些品牌列、有些只列巨量

**「客戶有時候需求來我們比對來不及」**——這是藥師朋友的原話（「客戶」指他藥局的客戶）。櫃檯前有人等著挑哪罐，肉眼掃成分表很容易漏行。工具的**第一價值**就是：統一格式、數值並排、差異可視化，手機上 3 秒內搞定。

#### 2. 實務面：市面上這種圖極度稀有，OCR 難，務實做法是親自拍

想直接爬網路商品頁的圖？不切實際：
- 賣家放 100 張圖，可能只有 10 張真的是成分標籤（其他是行銷圖、使用情境、禮盒照）
- 營養標示印刷字級很小，某些還印在罐身曲面
- 遠距/低解析/反光下的 OCR 準度差

**所以資料進入的務實做法**：
1. 親自到現場（家樂福、大樹）拍包裝
2. 拍時就要對焦、足夠光線、垂直鏡頭
3. 照片丟給 Claude / Gemini 辨識
4. AI 輔助整理成 `drinks.js` 的 schema

這個工具**不是爬蟲**，是「人 + AI 擷取 + 網站比對」的流程整合。

### 新增一筆資料的流程

這是本專案**最重要的工作流**，brain file `~/.claude/projects/-home-tom/memory/brain/health-drinks-data-entry.md` 有完整規則：

1. 買到一瓶新的（或找到有興趣的產品包裝）
2. **拍下正反面**成分標籤（重點是營養成分表）
3. 丟給 Claude/Gemini 辨識，要求「全填包含微量元素，不省略」
4. AI 產出一個符合 `drinks.js` schema 的 entry：
   ```js
   {
     id: "abbott-ensure-hmb-original",
     name: "安素 HMB 升級配方（原味）",
     brand: "亞培 Abbott",
     volume_ml: 237,            // 每份(沖泡後)體積,粉要換算成沖調後體積
     image: "images/亞培安素HMB.jpg",
     categories: ["醫療營養品"],
     nutrients: {
       "熱量(kcal)": 262,
       "蛋白質(g)": 11,
       "糖(g)": 14,
       "鈉(mg)": 150,
       // ... 標籤有什麼就填什麼
       "維生素A(mcgRE)": 200,   // 命名規範見 brain file
       "碘(mcg)": 30,
       "硒(mcg)": 14,
       // 微量元素全填
     }
   }
   ```
5. 把圖片放 `docs/images/`，entry 放進 `docs/data/drinks.js` 的 `DRINKS` 陣列
6. 瀏覽器打開網站確認新品項有出現在表格、比較矩陣

---

## 資料以「每份」儲存，前端換算 per 100ml

**重要**：`nutrients` 欄位值是**每份**（例 237ml 安素 = 262 kcal），不是 per 100ml。網站透過 `volume_ml` 做換算讓不同容量的產品公平比較。

粉狀產品的 `volume_ml` 要填**沖調後**體積（例 25g 粉 + 200ml 水 → `volume_ml: 225`）。

---

## 功能

- **排行表格**：依熱量、蛋白質、糖、鈉、鈣…排序
- **每 100ml 換算**：公平比較不同容量的產品
- **比較矩陣**：選幾瓶並排看差異，格子自動顏色標記（綠=優、紅=劣），左欄項目動態決定
- **圖片對照**：拍下來的產品包裝照
- **搜尋/篩選**：按品牌、分類

## 關鍵設計原則（來自 brain file）

1. **「加進去」= 全填含微量元素**：使用者叫 AI 補新資料時，只填巨量營養素會讓比較矩陣欄位缺失
2. **每份儲存，不是 per 100ml**：前端換算，否則雙重換算會錯
3. **粉狀 `volume_ml` 是沖泡後體積**：不是粉末重量
4. **欄位命名嚴格**：`熱量(kcal)` / `蛋白質(g)` / `鈉(mg)` / `碘(mcg)` / `維生素A(mcgRE)` / `菸鹼素(mgNE)`
